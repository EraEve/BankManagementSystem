## 1. 系统概述

银行智能管理系统是一个综合性的银行管理软件，完整实现了银行日常运营所需的全部核心功能模块。系统采用 **Python Flask 作为Web后端**，**Bootstrap 5 + Vanilla JS 作为SPA前端**，核心算法沿袭 **C++ 数据结构实现**，形成前后端分离的全栈Web应用。

### 1.1 核心特性

| 特性 | 说明 |
|------|------|
| **10大需求模块** | 职员管理、客户管理、银行卡管理、存取贷业务、业务查询、排队管理、网点导航、智能管理、身份核验、统一登录 |
| **52个REST API** | Flask后端提供完整RESTful接口，JSON数据交换 |
| **15种数据结构/算法** | 哈希表、链式队列、BST、邻接矩阵图、Dijkstra最短路径、冒泡排序、快速排序、二分查找、滑动窗口、加权多因子模型、ISO 7064校验、生物特征哈希、模拟OCR、Timsort、PBKDF2-SHA256 |
| **三级权限体系** | 管理员（全部功能）、职员（业务操作）、客户（自助查询） |
| **文件持久化** | 全部数据以管道分隔符文本文件存储，与C++版本完全兼容 |
| **SPA单页应用** | Bootstrap 5响应式界面，零框架依赖，动画过渡 |
| **Chart.js可视化** | 柱状图、环形图、投资配置图，动态数据仪表盘 |
| **DeepSeek AI** | 集成DeepSeek API提供智能投资分析，自动降级规则引擎 |
| **用户注册** | 自助注册功能，自动创建sdufe银行卡，注册即登录 |
| **前后端分离部署** | GitHub Pages（前端）+ PythonAnywhere（API）+ Render（全栈） |

### 3.2 安装步骤

```bash

git clone git@github.com:EraEve/BankManagementSystem.git
cd BankManagementSystem

# 2. 安装依赖

# 3. 启动服务
python app.py
# 浏览器自动打开 http://127.0.0.1:5000
```

如果需要在本地运行C++控制台版本：

```bash
# 编译
g++ main.cpp -o bank_system -std=c++11

# 运行
./bank_system        # Linux/macOS
bank_system.exe      # Windows
```

### 模块1: 银行职员管理

**菜单路径**: 管理员登录 → 导航栏 → 职员管理

**权限**: 仅管理员

**API端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/employees` | 获取所有职员列表 |
| POST | `/api/employees` | 添加新职员 |
| PUT | `/api/employees/<eid>` | 修改职员信息 |
| DELETE | `/api/employees/<eid>` | 删除(停用)职员 |
| PUT | `/api/employees/<eid>/password` | 重置职员密码 |

**操作说明**:

| 操作 | 说明 | 输入示例 |
|------|------|---------|
| 查看职员 | 表格展示所有职员，含状态标签 | 点击导航"职员管理" |
| 添加职员 | 编号格式 E+3位数字 | E003 → 姓名 → 密码(6位) → 角色 → 部门 → 电话 → 邮箱 |
| 删除职员 | 停用而非物理删除，管理员不可删除 | 点击行尾删除按钮 → 确认 |
| 修改职员 | 模态框编辑，空字段保留原值 | 修改需要的字段 → 保存 |
| 重置密码 | 管理员强制重置为6位新密码 | 点击重置 → 输入新密码 |

**数据结构**: `vector<Employee>` 顺序表 + `map<string,int>` 哈希索引 (O(1)查找)

**数据文件**: `data/employee.txt`

### 模块2: 客户账户管理

**菜单路径**: 管理员/职员登录 → 导航栏 → 客户管理

**权限**: 管理员和职员

**API端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/customers` | 获取所有客户列表 |
| POST | `/api/customers` | 添加新客户 |
| GET | `/api/customers/<cid>` | 获取单个客户详情 |
| PUT | `/api/customers/<cid>` | 修改客户资料 |
| DELETE | `/api/customers/<cid>` | 注销客户 |

**客户字段说明**:

| 字段 | 格式 | 说明 |
|------|------|------|
| 编号 | C+6位数字 | 系统自动分配，如 C000001 |
| 姓名 | 中文/英文 | 客户真实姓名 |
| 密码 | 6位数字 | 登录密码（PBKDF2-SHA256存储） |
| 类型 | 普通/VIP | 资产≥100,000自动升级为VIP |
| 电话 | 11位手机号 | 联系电话 |
| 开户日期 | YYYY-MM-DD | 系统自动填入当日 |
| 信用评分 | 0-1000 | 初始600，随交易调整 |
| 金融资产 | 金额(¥) | 客户总资产 |
| 地址 | 文本 | 通讯地址 |
| 身份证号 | 18位 | 中国居民身份证号 |

**VIP自动判定**: 当客户金融资产≥100,000元时，系统自动将客户类型升级为"VIP"。

**广义表关联**: 每个客户通过card_ids列表关联名下多张银行卡。

**数据结构**: `vector<Customer>` + `map<string,int>` 哈希索引 + BST二叉搜索树

**数据文件**: `data/customer.txt`

### 模块3: 银行卡管理

**菜单路径**: 管理员/职员登录 → 导航栏 → 银行卡管理

**权限**: 管理员和职员

**API端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/cards` | 获取所有银行卡列表 |
| POST | `/api/cards` | 添加新银行卡 |
| PUT | `/api/cards/<cid>` | 修改银行卡信息 |
| DELETE | `/api/cards/<cid>` | 注销银行卡 |

**三种银行卡类型**:

| 类型 | 特点 | 余额计息 | 信用额度 | 适用场景 |
|------|------|---------|---------|---------|
| 借记卡 | 先存后支 | 按余额×利率/365 | 无 | 日常消费、工资卡 |
| 储蓄卡 | 储蓄增值 | 按余额×利率/365 | 无 | 长期储蓄 |
| 信用卡 | 先消费后还款 | 按借贷余额×利率/365 | 有 | 透支消费 |

**sdufe学生卡**: 学生用户注册时自动生成格式为 `sdufe{学号}1` 的储蓄卡，初始余额0元，利率1.50%。

**数据结构**: `vector<BankCard>` + `map<string,int>` 哈希索引

**数据文件**: `data/card.txt`

### 模块4: 存取贷业务管理

**菜单路径**: 管理员/职员登录 → 导航栏 → 存取贷业务

**权限**: 管理员和职员

**API端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/deposit` | 存款 |
| POST | `/api/withdraw` | 取款 |
| POST | `/api/transfer` | 转账 |
| POST | `/api/loan` | 贷款(信用卡) |
| POST | `/api/repay` | 还款(信用卡) |

**各业务详细说明**:

#### 存款
1. 选择银行卡号 → 输入存款金额
2. 系统校验：卡状态正常
3. 卡余额增加，客户金融资产同步增加
4. 自动记录交易（类型：存款）

#### 取款
1. 输入银行卡号 → 输入金额
2. **三重校验**: 卡状态正常 → 余额充足 → 不超过日累计限额
3. 取款失败时自动记录失败原因
4. 成功后余额和资产同步减少

#### 转账
1. 输入转出卡号 → 转入卡号 → 金额
2. **安全校验**: 两张卡均正常 → 不能同卡转账 → 转出卡余额充足
3. 转出卡减、转入卡增
4. 自动记录两笔关联交易

#### 贷款（仅限信用卡）
1. 输入信用卡号 → 贷款金额
2. 校验：可用额度 = 信用额度 − 当前借贷余额
3. 贷款金额不得超过可用额度

#### 还款（仅限信用卡）
1. 输入信用卡号 → 还款金额
2. 校验：还款金额不超过借贷余额，且不超过卡内余额

#### 利息计算
- **日利息** = 本金 × (年利率%/100) / 365
- **月利息** = 本金 × (年利率%/100) / 12
- 储蓄卡/借记卡按余额计息，信用卡按借贷余额计息

**数据结构**: `vector<Transaction>` 顺序表

**数据文件**: `data/transaction.txt`

| 查询维度 | 参数 | 说明 |
|---------|------|------|
| 交易类型 | type | 存款/取款/转账/贷款/还款 |
| 日期范围 | date_from, date_to | 按交易时间范围筛选 |
| 金额区间 | amount_min, amount_max | 按交易金额范围筛选 |
| 卡号 | card_id | 精确匹配转出卡或转入卡 |
| 客户ID | customer_id | 按客户筛选 |
| 排序 | sort_by | amount(金额) / time(时间) |

**业务汇总统计** — 显示各类型交易的笔数和总金额，以表格和图表呈现。

**排序算法**:
- **冒泡排序**: 按交易时间排序 (O(N²)，展示基础算法)
- **快速排序/Timsort**: 按交易金额排序 (O(N log N))

### 模块6: 银行排队管理

**菜单路径**: 管理员/职员登录 → 导航栏 → 排队管理

**权限**: 管理员和职员

**API端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/queue/take` | 客户取号 |
| POST | `/api/queue/call` | 叫号办理 |
| GET | `/api/queue/status` | 查看排队状态 |
| GET | `/api/queue/history` | 历史排队记录 |
| GET | `/api/queue/stats` | 每日客流统计 |

**排队机制**:

```
       客户到达
           │
     ┌─────┴─────┐
     ▼           ▼
  VIP队列     普通队列
  (优先服务)   (顺序服务)
     │           │
     └─────┬─────┘
           ▼
    ┌──────────────┐
    │  VIP窗口(0)  │ ← 优先从VIP队列取
    │  普通窗口(1) │ ← 先查VIP队列，再查普通队列
    └──────────────┘
           │
           ▼
      办理完成 → 客户评分(1-5分)
           │
           ▼
      每日客流统计
```

**VIP优先策略**:
- VIP窗口: 先处理VIP队列，VIP队列空时才处理普通队列
- 普通窗口: 先查看VIP队列是否有等待（跨窗口服务），再处理普通队列
- 排队队列持久化，重启不丢失

**核心数据结构**: `LinkedQueue<T>` 链式队列（自行实现）

**数据文件**: `data/queue.txt`

#### 8.1 交易异常报警

**大额交易检测**: 扫描所有交易，标记金额 > 50,000元的交易记录。

**高频交易检测**（滑动窗口算法）:
- 按客户分组统计交易
- 滑动窗口扫描同一小时内交易笔数
- 超过5笔/小时触发报警

#### 8.2 批量利息计算

遍历所有正常状态银行卡，计算日利息/月利息并汇总。

#### 8.3 客户统计分析 & 信用评级

**统计维度**: 活跃客户数、VIP/普通占比、平均/最高/最低金融资产、信用评分分布

**信用评级模型**（加权多因子）:

| 因子 | 权重 | 说明 |
|------|------|------|
| 金融资产 | 30% | 资产越多评分越高 |
| 交易活跃度 | 20% | 正常交易频率 |
| 还款记录 | 25% | 按时还款加分 |
| 负债率 | 25% | 负相关（负债越高扣分越多） |

**五级评级**:

| 评分范围 | 等级 | 含义 |
|---------|------|------|
| 900+ | AAA | 极优 |
| 800-899 | AA | 优秀 |
| 700-799 | A | 良好 |
| 600-699 | B | 一般 |
| <600 | C | 需关注 |

#### 8.4 风控审批

综合评估维度: 客户信用评分 + 负债率 + 金融资产

**审批结论**:
- 风控分数 ≥ 700: ✓ 通过（低风险）
- 风控分数 ≥ 600: ⚡ 条件通过（中风险，建议降低额度）
- 风控分数 < 600: ✗ 拒绝（高风险）

#### 8.5 投资顾问

```
Sum = Σ(第i位数字 × 权重[i])
校验码映射 = ['1','0','X','9','8','7','6','5','4','3','2']
expected = 校验码映射[Sum % 11]
```

## 14. 常见问题

### Q1: 启动报错 "ModuleNotFoundError: No module named 'flask'"

**解决**: `pip install -r requirements.txt`

### Q2: 登录后页面空白/API报401

**原因**: Session未正确传递（跨域问题）。
**解决**: 
- 确保前端和后端同源访问（本地都用 localhost:5000）
- 部署环境下CORS已自动配置

### Q3: AI投资分析不工作

**原因**: DeepSeek API Key未配置。
**表现**: 自动降级为规则引擎模式，功能正常使用。
**配置**: 设置环境变量 `DEEPSEEK_API_KEY` 或在WSGI文件中配置。

### Q4: 数据文件被误删了怎么办？

删除 `data/` 目录下所有文件，重新运行程序，系统会自动创建默认数据。

### Q5: 如何重置管理员密码？

**Web端**: 管理员登录后通过职员管理 → 重置密码。
**手动**: 编辑 `data/employee.txt`，找到管理员行（E001），修改第三个字段。

### Q6: Web端和C++端的数据互通吗？

数据格式完全兼容（管道分隔符），但运行时数据独立。Web端使用 `data/account.txt` 存储学生账户，C++端不使用该文件。

### Q7: 如何添加新学生账号？

**方式1**: 使用注册功能，学生自助注册（v3.3.4+）。
**方式2**: 管理员登录 → 客户管理 → 添加客户。
**方式3**: 直接编辑 `data/account.txt` 文件添加。

### Q8: PythonAnywhere部署后前端连不上API？

检查：
1. CORS配置：确保前端域名在白名单中
2. Session Cookie: PythonAnywhere自动启用 `SameSite=None`
3. WSGI文件：确认 `app` 对象正确导出

### Q9: 中文显示乱码

- 确保所有 `.txt` 数据文件为 UTF-8 编码
- 浏览器设置编码为 UTF-8
- C++控制台: 运行 `chcp 65001` 切换到UTF-8

### D. 安全性说明

系统实施了6层安全防护：
1. **密码哈希**: PBKDF2-SHA256，明文密码自动迁移
2. **Session鉴权**: HttpOnly Cookie + 服务端Session
3. **CORS白名单**: 正则匹配允许的域名
4. **SameSite Cookie**: 生产环境 None（跨域） / 本地 Lax
5. **Secure Cookie**: 生产环境仅HTTPS传输
6. **SECRET_KEY固化**: 环境变量 → 文件 → 自动生成，防止Session丢失

### E. 退出与数据保存

**Web端**: Session自动过期。所有业务操作即时持久化到 `data/` 目录。

**C++端**: 在主界面选择 `0. 退出系统`，系统自动保存所有数据。

**保存的数据包括**:
- 账户数据 (account.txt)
- 职员数据 (employee.txt)
- 客户数据 (customer.txt)
- 银行卡数据 (card.txt)
- 交易记录 (transaction.txt)
- 排队历史 (queue.txt)
- 网点数据 (branch.txt)
